# 代码框架与技术栈选型 (Architecture & Tech Stack)

为了达成“快速开发、免费部署、无需持久化数据库”的聚会桌游平台目标，我们的技术栈选择了**最容易上手且目前工业界极其成熟的轻量化方案**。

---

## 1. 总体架构哲学 (Zero-Database Architecture)
- **无状态前端客户端**：前端像是一个轻量级的“遥控器+屏幕”，只负责渲染漂亮的游戏盘面和发送玩家的点击操作。不参与判定输赢。
- **内存型服务端 (In-Memory)**：完全砍掉 MySQL 等数据库引擎。**一切游戏状态、谁进入了什么房间，全放在 Node.js 进程的内存字典 (Map) 里。**
- **短择效应**：服务器一旦由于免费套餐限流发生重启，当前的牌局将丢失。但在朋友聚会玩小桌游的场景中，大不了“重新开一盘”就好。换来的是极其低廉的开发与维护成本。

---

## 2. 前端选择 (Frontend)

- **UI 渲染核心：React (配合 Vite 工具链)**
  - 非常适合用于搭建桌游平台。桌游中的卡牌、筹码、化身头像可以非常容易地被抽象成各种 React 组件复用。
- **样式方案：Tailwind CSS**
  - **绝配**。想要那种 Apple/Google 风的极简扁平化、高对比效果，Tailwind 是一把利器。可以在不写额外 CSS 文件的情况下迅速调整按钮圆角、投影、文字间距。
- **路由方案：React Router**
  - 管理在 `主页` -> `大厅` -> `游戏中` 的平滑过渡。
- **网络层：Socket.io-client**
  - 自动处理长连接和轮询回退，非常省心。

---

## 3. 后端选择 (Backend)

- **核心运行环境：Node.js (建议用 TypeScript 增加代码防呆性，如果求快也可以纯 JS)**
  - 聚会游戏的频次不高，不需要太极端的并发能力，JS 的异步事件流处理机制非常切合这种大厅/房间的网络响应模式。
- **通信库：Socket.io**
  - 为什么不直接用纯 WS？因为 **Socket.io 默认自带了 Room (房间) 管理机制**。
  - 你可以直接用代码 `socket.join('ROOM_1234')`，然后直接广播 `io.to('ROOM_1234').emit('message')`，它自动帮你处理向该房间全体发包的事宜。这对基于房间的系统来说是最大的福音，直接节省了造轮子的时间。

---

## 4. 前后端公共模块 (Shared Logistics)
由于没有数据库，可以通过在单一工程内采用 Monorepo（一种目录结构）或前后端共享常量的形式，确保前后端对于“游戏事件名称”（如 `EVENT_PLAYER_JOIN`, `EVENT_GAME_START`）和桌游的数据结构有一致的定义，避免打字手误带来的 Bug。
